### Overview
Pada postingan kali ini saya akan membahas bagaimana penerapan infrastructure as code bekerja,
infrastructure as code atau disingkat sebagai iac adalah salah satu bentuk metodologi dalam dunia automation 
sebagai salah satu contoh kamu bisa aja mengotomatisasi semua proses konfigurasi yang tadinya manual jadi otomatis tanpa manual 
handling lagi, use casenya misalkan kamu punya lebih dari 100 node server yang harus di konfigurasikan akan memakan waktu yang sangat
lama untuk melakukan konfigurasi satu persatu terhadap semua node yang ada,maka dari itu lahirlah solusi yaitu automation iac,
dengan iac kamu bisa melakukan konfigurasi ke seluruh node yang ada tanpa harus melakukan manual handling satu persatu.Jadi iac lahir 
sebagai solusi dan iac juga bisa di terapkan ke semua model komputasi baik on-premise , cloud , ataupun container, langsung saja
masuk ke pembahasanya

```
sudo dnf install ansible-core
```


jika keluar output seperti ini

maka ansible sudah berhasil di install di node **INGAT ANSIBLE ITU AGENTLESS JADINYA HANYA PERLU SATU ANSIBLE TIDAK PERLU INSTALL
DI SEMUA NODE YANG AKAN MENJADI WORKER**.jika sudah masuk ke tahap selanjutnya yaitu menyiapkan inventory dan ansible configuration.
Untuk tahap selanjutnya yaitu membuat ansible config 

```
vim ansible.cfg
|
[iki@node1 ~]$ cat ansible.cfg
[defaults]
inventory = ./inventory
remote_user = iki
host_key_checking = False
become = yes
become_method = sudo
become_ask_pass = False
```

ansible config berfungsi untuk memberitahu ansible lokasi dari inventory tapi bukan hanya itu saja ada beberapa fungsi lain dari ansible config ini seperti remote user , privillage esallation , misalnya untuk remote user ketika akan menggunakan ansible cara kerja ansible itu menggunakan ssh sebagai cara kerja dari si ansible,
nah untuk remote user sendiri itu bisa di tentukan lewat ansible config dengan remote user,selain dari itu misalkan privillage escallation penting
untuk mengeksekusi task dan modul yang ada karena ada banyak task yang membutuhkan root permission jika saja tidak ada privillage escalltion 
tidak mungkin task yang memerlukan root permission bisa di eksekusi, **NOTE JIKA MENGGUNAKAN PRIVILLAGE ESCALATION PASTIKAN REMOTE USER YANG ADA DI TARGER SERVER SUDAH MASUK KE FILE SUDOERS!! GUNAKAN PERINTAH SUDO VISUDO AGAR BISA MENGEDIT SUDOERS**,
jika sudah maka masuk ke tahap selanjutnya yaitu membuat inventory

```
vim inventory
|
[master]
node1 ansible_host=192.168.1.xxx

[slave1]
node2 ansible_host=192.168.1.xxx

[slave2]
node3 ansible_host=192.168.1.xxx



[all:vars]
[all:vars]
ansible_user=iki
ansible_ssh_private_key_file=~/.ssh/id_rsa

```

inventory berfungsi sebagai tempat penyimpanan informasi penting seperti grup dan hostname atau ip dari target server,
lihat contoh diatas ada baris dengan nama master , slave1,slave2 nah itu semua adalah grup yang bisa berisi ip dari target ataupun hostname dari target,
dan di inventory juga bisa menyimpan variable yang bisa di pakai kapan saja misal dalam eksekusi playbook,ingat dengan ansible config diatas 
ada baris dengan value seperti ini

```
./inventory
```

nah itu berarti inventory di simpan dan di pakai di direktori yang sama dengan si user misalkan user iki ada di /home/iki nah maka dari itu jika 
di ansible config inventory sudah di tentukan pathnya maka user harus membuat inventory sesuai dengan yang ada di ansible config.
Masuk ke tahap selanjutnya yaitu copy ssh key, pertama-tama buat terlebih dahulu ssh key nya gunakan command

```
ssh-keygen -t rsa
```

tekan enter sampai selesai agar key tetap dalam keadan default tanpa adanya editing, jika sudah maka masuk ke pembuatan playbook pertama yaitu
playbook ssh-copy

```
vim playbook-ssh.yaml
|
- name: copy ssh
  hosts: slave1,slave2,msater
  become: yes

  tasks:
    - name: copy ssh key
      authorized_key:
        state: present
        user: iki
        key: "{{ lookup('file' , '~/.ssh/id_rsa.pub') }}"
```

playbook diatas berfungsi sebagai playbook yang mengcopy semua ssh ke semua node yang ada di inventory mari bahas playbook pertama ini,
masih ingat dengan inventory yang
sebelumnya di buat semua node target sudah masuk inventory dengan ip masing masing dari node tersebut, dan di playbook pertama ini 
node target ada slave1,slave2,master semua hosts ini adalah node yang sudah masuk grup di inventory,jadinya tinggal gunakan saja nama dari grupnya
lalu ada become yes masih ingat dengan ansible config yang sudah dibuat nah bagian ini mengatur privillage escallation, karena ini task yang
ada di jalankan dengan root permission,nah untuk seterusnya isi dari playbook itu template jadi bagian atas dari playbook pertama itu sama nantinya dengan playbook lainnya,jadinya saya hanya menjelaskan bagian modul,
masuk ke modul selanjutnya,modul kali ini akan menonaktifkan swap secara permanent,swap mati adalah salah satu syarat utama buat kubelet bisa berjalan buat file dengan nama

```
- name: disable swap permanent
  hosts: slave1,slave2,slave3
  become: yes

  tasks:
    - name: disable swap
      ansible.builtin.lineinfile:
        regexp: '^/dev/mapper/rhel-swap'
        insertbefore: '^/dev/mapper/rhel-swap'
        line: '#/dev/mapper/rhel-swap'
        path: /etc/fstab
    - name: swap disable
      ansible.builtin.command: swapoff -a
```

playbook diatas merupakan playbook untuk membuat swap mati secara permanent,dengan mencari regex dari path file tertentu
